    03Decisiones6 bloqueadas
    #DecisiónEstado
    1Opción C (híbrido) — agregar solo campos de altísima confianza + opcionalizar operativos + documentar mapping. NO refactor al envelope consumers.creditReport[]. Confirmada por el usuario.◆ elegido
    2Shape plano se mantiene — BuroMockResponse sigue siendo nuestro "domain model" interno; el adapter futuro hace la traducción al envelope Equifax.◆ elegido
    3status siempre "completed" en mock. El type admite "pending"/"error" para forward-compat, pero el mock no los genera hasta que existan casos reales.◆ elegido
    4hitCode siempre { code: "1" } en mock. Shape fiel a Equifax US; "0" (no-hit) requiere modelar consumers no encontrados, no aplica con cédula CR válida determinística.◆ elegido
    5links siempre [] en mock. Cuando el cliente real devuelva links, se validan contra allowlist antes de cualquier fetch (CWE-918/SSRF).◆ elegido
    6Campos operativos del request se ignoran en mock — si se usaran como inputs, dos calls idénticas con distinto memberNumber tendrían que devolver respuestas distintas, rompiendo el determinismo por cédula.◆ elegido

    04Riesgos4 invariantes
    ▲
    CWE-532 · Logging de credenciales operativas. memberNumber/securityCode/customerCode son credenciales del cliente en Equifax. El log [buro] solo registra {ts, category, score}; los nuevos campos del request se descartan explícitamente. Test CWE-532 se extiende a los nuevos campos.
    ▲
    CWE-200 · Invariante PII (INV-1). Solo riskLevel (1/2/3) llega al LLM. Los nuevos campos (status, hitCode, links) se quedan en HTTP/engine y nunca se interpolan en SYSTEM_PROMPT_TEMPLATE ni en riskLevelInstruction.
    ▲
    CWE-918 · links[].href como URL no confiable. El type lo permite; cuando llegue el real, debe validarse contra allowlist de hosts de Equifax antes de cualquier fetch automático. El motor NO debe seguir links automáticamente; el caller decide. Mock siempre devuelve [].
    ▲
    Determinismo por cédula (CWE-20 inverso). Los campos operativos del request NO deben sesgar el resultado simulado. Dos calls con el mismo idNumber y distinto memberNumber devuelven respuesta idéntica. Test explícito en S5.
