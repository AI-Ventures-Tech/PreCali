PreCali → Next.js

Migración completa del sitio estático (HTML/CSS/JS) y del bot de WhatsApp (Vercel Serverless) a Next.js 15 App Router + React 19 + TypeScript + Tailwind 4.
Proyecto: PreCali Creado: 2026-07-01 Subtareas: 17 Fases: A · B · C · D Stack: TS · App Router · React 19 · Tailwind 4

Contexto

PreCali hoy es un sitio estático sin build: index.html + styles.css (57 KB) + 4 JS tipo IIFE que comparten estado por globales (PAISES, BANCOS, TIPO_CAMBIO_USD) y exponen handlers en window. para onclick inline.

Además existe un bot de WhatsApp en api/ como Vercel Serverless (CommonJS, ≈3700 LOC en api/_lib/) que usa ~20 variables de entorno (Twilio, Groq, OpenAI, Upstash KV).

Hallazgos clave: no hay package.json ni tests ni TS · el lead form está roto (posti­fica a / con el patrón Netlify Forms, que no funciona en Vercel) · vercel.json define cabeceras a migrar · solo Costa Rica está operativo.

Decisiones aprobadas

  • 1. Migración completa: frontend + reescritura del bot a Route Handlers.
  • 2. Stack TS + App Router + React 19 + Tailwind 4, preservando la estética original.
  • 3. Lead → Route Handler (zod + anti-spam); data → módulos TS tipados.

Invariantes de seguridad

CWE-79 — Ningún input de usuario/WhatsApp se renderiza sin escape. Se prohíbe dangerouslySetInnerHTML salvo con sanitización.
CWE-306 — El webhook de WhatsApp verifica la firma X-Twilio-Signature antes de procesar; si no coincide → 403 sin side effects.
CWE-770/799 — El endpoint de leads aplica honeypot + rate-limit por IP y rechaza payloads fuera del schema zod.
CWE-601/918 — Las URLs/medios de Twilio se validan contra allowlist de dominios antes de cualquier fetch (sin SSRF/open redirect).
CWE-89 — Los accesos a Upstash KV usan el SDK oficial; nunca concatenación de strings para claves.
CWE-532 — Ningún secreto se loguea ni se envía al cliente; el código que los toca vive bajo src/lib/**.
CWE-22 — Los paths temporales de OCR se validan dentro de tmpdir() sin ...
