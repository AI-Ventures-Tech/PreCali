Motor calificador con mock de buró

Clasifica cada lead del bot de WhatsApp en un nivel de riesgo 1/2/3 usando un mock determinístico de Equifax/SUGEF. El mismo idNumber siempre produce el mismo resultado. Sin acceso real al buró todavía.

    ●
    01Qué es y por quécontexto de negocio

    PreCali es un bot de WhatsApp que precalifica leads para créditos bancarios en Costa Rica. La Fase 3 del roadmap Zero-Friction pide clasificar cada lead en un nivel de riesgo para que el bot pueda:

    • Ajustar el tono del asesor conversacional (educativo para riesgo alto, comercial para prime).
    • Decidir el siguiente paso en el flujo: ofrecer banco tradicional, alternativa con más apetito de riesgo, o redirigir a un plan de saneamiento a 6 meses.

    No existe acceso real a la API de Equifax todavía. Este feature entrega el mock del buró + el motor de clasificación puro, dejando la integración real fuera de alcance.

    ●
    Fuentes del mock. Las 8 categorías reales del CIC de SUGEF (A1, A2, B1, B2, C1, C2, D, E) y el Comportamiento de Pago Histórico en 3 niveles, con historial de 48 meses. El portal developer.latam.equifax.com confirma OAuth2 + sandbox pero su JSON schema exacto está detrás de login.
    ▣
    02Pipeline del flujocédula → nivel → acciones

    El pipeline se dispara cuando el usuario envía nombre + cédula + email en el paso stepLeadDatos. Hasta ese momento el flujo es el mismo que antes; desde ahí en adelante, todo gateo depende del nivel calculado.

    Captura de cédula →
    flow.ts · stepLeadDatos
    El usuario manda su cédula en un mensaje. Una línea de consentimiento (Ley 9859) va en el mismo texto que la pide.
    Mock del buró →
    buro/mock-equifax.ts
    Hash FNV-1a de la cédula → PRNG mulberry32 → BuroMockResponse determinístico. Nunca Math.random ni Date.now dentro.
    Motor de reglas →
    buro/engine.ts · calificarLead()
    Aplica 5 reglas en orden fijo (mora → categoría SUGEF → score → ratio → cap) y devuelve EngineResult con nivel: 1|2|3.
    Persistencia en sesión
    whatsapp/types.ts · memory.ts
    session.buroResult guarda el resultado. Solo se persiste en memoria por now — el TTL de 90 días (Ley 8968) queda como gap explícito.
    ↓
    Gateo del flujo →
    flow.ts · goToHardPull
    Nivel 1 → mensaje de rescate + session.step = "pausado" (sin botón "Autorizo banco"). Nivel 2 → nota de prima/ajuste en el prompt de autorización.
    Tono del asesor IA
    whatsapp/agent.ts · nivelRiesgoInstruccion()
    El system prompt recibe solo el número de nivel (1/2/3). Nunca el score ni operaciones — invariante PII (CWE-200).
    ●
    Límite de alcance. La primera tabla de simulación de bancos se muestra antes de pedir la cédula, así que el nivel no puede gatearla sin un rediseño mayor. El nivel aplica desde stepLeadDatos en adelante (decisión confirmada con el usuario).

    04Motor: precedencia de reglas5 reglas en orden fijo

    El motor calificarLead(buro, profile) aplica las reglas en orden estricto. La primera que matchea fija el nivel. El orden no se puede reinterpretar — está fijado por negocio y validado en tests.

    #ReglaNivelNotas
    1Mora activa severa: cualquier operación con diasAtraso > 901Override duro. Pisa categoría y score, sin excepción.
    2Categoría SUGEF en {B2, C1, C2, D, E}1B2 explícitamente cuenta como alto riesgo (corregido por el usuario sobre el default inicial).
    3Categoría {A1, A2, B1} + score ≥ 700 + ratio deuda/ingreso ≤ 0.453Prime precalificado: umbrales estrictos en score y ratio.
    4Default para categoría {A1, A2, B1} que no cumple la regla 32Buen historial pero score o ratio insuficiente.
    5Cap: si ratio deuda/ingreso > 0.50 y el nivel resultante es 3 → baja a 22Marca ratioAlto = true. Nunca permite Nivel 3 con deuda alta.

    Flag lateral (no cambia el nivel): si entidadesConsultantesUltimos30Dias > 5 → shoppingCredito = true. Se reporta en EngineResult para logging/debug pero no participa en la decisión.

    ●
    Sanitización de config. mergeConfig() valida que cada umbral custom sea typeof number && Number.isFinite && ≥ 0. Un valor corrupto cae a DEFAULT_ENGINE_CONFIG en ese campo — un umbral roto no puede aprobar o rechazar leads silenciosamente (CWE-20).

    Defaults (DEFAULT_ENGINE_CONFIG)

    scorePrimeThreshold:         700    // umbral "prime"
ratioDeudaIngresoAlerta:     0.50   // arriba de esto → cap a Nivel 2
ratioDeudaIngresoPrecalificado: 0.45 // tope para Nivel 3
moraActivaDiasLimite:        90     // >90 días → Nivel 1 duro
shoppingCreditoConsultas30d: 5      // >5 consultas → flag (no cambia nivel)

    05Mock del buró (determinístico)hash FNV-1a + mulberry32

    El generador convierte la cédula en una respuesta de buró reproducible. La misma cédula produce siempre el mismo resultado byte a byte — clave para demos y tests estables.

    Cadena de derivación

    idNumber
  │
  ├─ fnv1a(idNumber)              → semilla entera de 32 bits (hash aritmético puro)
  │
  └─ mulberry32(seed)             → PRNG () => float en [0, 1)
        │
        ├─ pickCategoria()        → una de 8 categorías SUGEF (distribución ponderada)
        ├─ score                  → randInt en [scoreMin, scoreMax] de la categoría
        ├─ operaciones[]          → 0-4 operaciones, monto/días coherentes con la categoría
        ├─ montoTotalAdeudado     → suma de operaciones
        ├─ consultas30d           → sesgadas a 0, con bump para categorías de riesgo
        └─ protestosComerciales   → probabilidad por categoría

    Las categorías malas (C1/C2/D/E) tienen moraGarantizada: true: si el PRNG no produjo ninguna mora activa, se fuerza al menos una sobre una operación existente. Así el mock es coherente con la regla 1 del motor.

    Distribución de categorías (pesos realistas)

    CategoríaPesoScoreMora garantizadaNotas
    A15%800–850nocola prime
    A218%740–810nomasa central
    B122%680–750nomasa central
    B220%620–690nomasa central · mapea a Nivel 1
    C114%540–630sícola de riesgo
    C210%460–550sícola de riesgo
    D7%380–470sícola de riesgo
    E4%300–390sícola de riesgo

    Pesos elegidos para que la mayoría de cédulas de prueba caigan en B1/B2 (lo más común en población real). 200 cédulas generan ≥3 categorías distintas — test de sanity en mock-equifax.test.ts.

    06Invariantes de seguridad4 reglas no negociables
    ✕
    CWE-200 · PII financiera. La cédula (idNumber) y el payload crudo del buró (score, operaciones, protestos) nunca se envían a Groq. El system prompt del asesor solo recibe el nivel derivado + flags booleanos. Verificado por test agent.test.ts que aserta que ningún campo crudo aparece en el prompt.
    ▲
    CWE-20 · Validación de config. mergeConfig() rechaza umbrales NaN/undefined/negativos y cae a defaults. Un umbral corrupto no puede aprobar o rechazar leads silenciosamente.
    ▲
    CWE-77/78 · Inyección. El hash que siembra el PRNG es puramente aritmético sobre el string del idNumber. Sin eval, sin interpolación en plantillas o comandos shell.
    ▲
    CWE-359 · Retención de PII. No se persiste BuroMockResponse ni EngineResult fuera de la sesión en memoria. El TTL/purga de 90 días (Ley 8968) queda como gap documentado — requiere una capa de persistencia que no existe aún. Ver Needs Attention.

    flow.ts · stepLeadDatos — cálculo del nivel

    flow.ts · goToHardPull — gateo por nivel

    if (session.buroResult?.nivel === 1) {
  // Nivel 1: NO se ofrece "Autorizo banco". Mensaje de rescate + pausa.
  return {
    actions: [actionTexto("...plan de saneamiento a 6 meses...")],

    08Decisiones de negocio confirmadas4 + 2 no bloqueantes

    Decisiones que moldearon el diseño

    #DecisiónRazón
    D1B2 → Nivel 1 (alto riesgo), no Nivel 2.Corregido por el usuario sobre el default. Una categoría B2 debe tratarse como riesgo alto, igual que C1-E.
    D2El mock se dispara justo después de capturar la cédula, sin tocar el paso OCR existente.Mantiene el paso de OCR (stepAutorizarSoftPreCali) intacto. Son dos cosas distintas.
    D3El mock es determinístico por cédula.Mismo idNumber → mismo resultado. Reproducible en pruebas y demos.
    D4El nivel no gatea la primera tabla de bancos.El banco se elige antes de pedir la cédula. Gatear la primera oferta requeriría mover la captura arriba — rediseño mayor, fuera de alcance.

    Preguntas no bloqueantes (confirmadas para esta fase mock)

    ◆
    Q1 · Consentimiento (Ley 9859 art. 44 bis). Una sola línea de texto en el mensaje que pide la cédula, sin paso de botón extra. Preserva la fricción cero. Se revisará con abogado cuando llegue el acceso real al buró.
    ◆
    Q2 · Escala de score 300-850, umbral prime 700. Placeholder razonable (convención tipo FICO/VantageScore). Equifax CR/LATAM no publica su schema sin login. Se recalibra cuando exista sandbox real.
